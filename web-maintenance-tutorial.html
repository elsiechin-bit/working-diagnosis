1. From 1990s HTML to Today

Welcome! Your 1990s HTML knowledge is actually foundational. The basic idea hasn't changed — HTML is still markup, tags still work the same way. But the ecosystem has evolved. Let me show you what's different and what's stayed the same.

What's the Same

These tags still work exactly like you remember:

<p>Paragraph</p>
<h1>Heading</h1>
<ul><li>List item</li></ul>
<a href="url">Link</a>
<img src="image.jpg" alt="text">

What's Different

Then (1990s) Now (2020s)
Hand-written HTML files Static site generators like Eleventy
Inline CSS in <style> tags Separate CSS files, or CSS variables
Tables for layout CSS Grid and Flexbox for layout
JavaScript for hover effects CSS for interactivity; JS for complex behavior
FTP to deploy Git and GitHub for version control, automated deployment

The Big Picture

Here's the workflow for the Working Diagnosis site:

  1. Write content in Markdown (human-friendly format)
  2. Run Eleventy build (converts Markdown + templates → HTML)
  3. Preview locally in your browser
  4. Commit to git (saves a snapshot of changes)
  5. Push to GitHub (uploads to the repository)
  6. GitHub Actions deploys (automatically publishes the site)
Good news: You don't need to write HTML by hand anymore. Eleventy generates it from your Markdown content. Your job is to edit Markdown files and understand how the build process works.

Tools You'll Use

  • VS Code — Text editor (free, modern, very good)
  • Terminal / PowerShell — Command line interface
  • Git — Version control system
  • Node.js — JavaScript runtime (runs Eleventy)
  • npm — Package manager (comes with Node.js)

You probably have most of these already if the site is set up locally. In the next section, we'll look at modern HTML structure.

2. Modern HTML & Semantics

Modern HTML added semantic tags that describe what content means, not just how it looks. This is huge for accessibility, SEO, and maintainability.

Semantic HTML Elements

Instead of <div> everywhere, use tags that describe the content:

<!-- OLD WAY (1990s) -->
<div class="header">
    <h1>Site Title</h1>
    <div class="nav">
        <a href="/">Home</a>
    </div>
</div>

<!-- NEW WAY (Semantic) -->
<header>
    <h1>Site Title</h1>
    <nav>
        <a href="/">Home</a>
    </nav>
</header>

Key Semantic Tags

Tag Purpose
<header> Page or section header (logo, title, nav)
<nav> Navigation links
<main> Main content area (only one per page)
<article> Self-contained content (blog post, news article)
<section> Thematic grouping of content
<aside> Sidebar or supplementary content
<footer> Page footer (copyright, links)

Attributes: id and class

id and class let you target HTML elements with CSS and JavaScript:

<!-- id: unique identifier for ONE element -->
<header id="site-header">...</header>

<!-- class: reusable label for multiple elements -->
<p class="intro-text">...</p>
<p class="intro-text">...</p>

<!-- Multiple classes -->
<button class="btn btn-primary btn-large">Click me</button>
Example: Typical article page structure 
<header>
    <h1>Site Name</h1>
    <nav>...</nav>
</header>

<main>
    <article>
        <h1>Article Title</h1>
        <p>Article content...</p>
    </article>

    <aside>
        <h2>Related Links</h2>
        <ul>...</ul>
    </aside>
</main>

<footer>
    <p>© 2026 My Site</p>
</footer>

The Working Diagnosis Site

The site is built using Eleventy templates (Nunjucks), which generate semantic HTML like this. You rarely write HTML by hand; instead, Eleventy generates it from:

  • Markdown files (your content)
  • Templates (structure and styling)
  • Config files (settings)

In the next section, we'll look at CSS — how to style that HTML.

3. CSS: Styling the Web

CSS (Cascading Style Sheets) controls appearance. Modern CSS is much more powerful than 1990s CSS — no more tables for layout!

CSS Basics: Selectors

A CSS rule has three parts: selector, property, value.

p {
    color: blue;        /* property: value */
    font-size: 16px;
}

.intro {
    background: yellow;
}
Selector Matches
p All <p> elements
.intro All elements with class="intro"
#header The element with id="header"
header p All <p> inside <header>

Modern Layout: Flexbox and Grid

Instead of tables, use Flexbox (one dimension) or Grid (two dimensions):

Flexbox (arrange items in a row or column)

.container {
    display: flex;
    gap: 20px;          /* space between items */
    justify-content: center;  /* horizontal alignment */
}

.item {
    flex: 1;            /* equal width */
}

Grid (arrange items in rows and columns)

.grid {
    display: grid;
    grid-template-columns: 1fr 2fr 1fr;  /* three columns */
    gap: 20px;
}

CSS Variables (Custom Properties)

Define colors, sizes, and fonts once, use them everywhere:

:root {
    --primary-color: #667eea;
    --text-color: #333;
    --spacing: 20px;
}

p {
    color: var(--text-color);
    margin-bottom: var(--spacing);
}

button {
    background: var(--primary-color);
}
Why this matters for Working Diagnosis: The site uses CSS variables for colors and spacing. When you change a color in the CSS variable, it updates everywhere automatically. No hunting for hardcoded colors.

Responsive Design (Mobile-First)

CSS media queries adapt layout for different screen sizes:

/* Default: mobile layout */
.grid {
    grid-template-columns: 1fr;  /* one column on small screens */
}

/* On larger screens: */
@media (min-width: 768px) {
    .grid {
        grid-template-columns: 2fr 1fr;  /* two columns */
    }
}

Debugging CSS

Right-click on any page → Inspect to open developer tools. You can:

  • See the HTML structure
  • Read applied CSS rules
  • Change styles live (not saved) to test
  • Check responsive layout at different sizes
Try this:
  1. Right-click this page → Inspect
  2. Find a paragraph (<p>)
  3. In the Styles panel, see what CSS is applied
  4. Click a color value to change it (testing only)

4. Command Line Basics

The terminal (command line) is where you'll run build commands, manage git, and navigate your computer's file system. Don't worry — you only need a few commands.

Opening the Terminal on Windows

  1. Open File Explorer
  2. Navigate to your project folder (working-diagnosis)
  3. Right-click in empty space → Open in Terminal or Open PowerShell here

Or use the keyboard shortcut: Win + R, then type powershell.

Essential Commands

Command What it does
pwd Print working directory (where am I?)
ls or dir List files in current folder
cd folder-name Change directory (navigate into a folder)
cd .. Go up one level
mkdir folder-name Create a new folder
rm filename Delete a file
clear Clear the screen

Project-Specific Commands

These run in your project folder:

npm run build     # Run the build process
npm start         # Start the local development server
npm run build:nopdf  # Build without generating PDFs

Understanding Command Output

When you run npm run build, you'll see text scroll by. Here's what to look for:

Success: 
[11ty] Writing _site/index.html
[11ty] Copied 147 files
✓ Build completed
Error: 
Error: Template render error
  undefined variable "title" in page.njk:5

Check your front matter or template syntax.
If you see an error:
  1. Read the error message carefully
  2. It usually says which file has the problem
  3. Check that file for typos or missing fields
  4. Run build again after fixing

Stopping a Running Process

If the development server is running (from npm start) and you want to stop it, press:

Ctrl + C

Next: let's learn git, which tracks changes to your files.

5. Git & Version Control

Git is version control — it tracks every change you make to your code. It's how you save snapshots (commits) and collaborate. GitHub is where your code lives online.

The Core Workflow

  1. Edit files (content, styling, etc.)
  2. Stage changes (tell git what to save)
  3. Commit (create a snapshot with a message)
  4. Push (upload to GitHub)

Essential Git Commands

Command What it does
git status Show what's changed since last commit
git add . Stage all changes
git commit -m "message" Create a snapshot with a message
git push origin main Upload commits to GitHub
git log --oneline See past commits

Step-by-Step: Making a Change

# 1. Check what changed
git status

# 2. See the differences
git diff

# 3. Stage everything
git add .

# 4. Commit with a message (present tense)
git commit -m "fix typo in heart-failure page"

# 5. Upload to GitHub
git push origin main

Good Commit Messages

Write clear, concise messages in present tense:

✅ Good ❌ Bad
fix typo in AF page stuff, fixed things
add hypertension guidelines updates, changes
improve mobile menu contrast WIP, temp, asdf

Viewing History

See all your past commits:

git log --oneline

Output looks like:

a1b2c3d (HEAD -> main) fix typo in heart-failure page
f4e5d6c add hypertension screening section
7g8h9i0 update AF guidelines

Undoing Changes

If you mess up, you can undo:

# Undo uncommitted changes to a file
git checkout filename

# Undo the last commit (keep changes)
git reset --soft HEAD~1

# Undo everything since last commit
git reset --hard HEAD
Be careful with reset --hard — it deletes unsaved work. Only use if you're sure.

If Git Gets Stuck

Sometimes git files lock up. Run this PowerShell command in your project folder:

Get-Process | Where-Object {$_.ProcessName -like "*git*"} | Stop-Process -Force; Remove-Item -Force .git/*.lock -ErrorAction SilentlyContinue

6. Static Sites & Eleventy

Eleventy is a static site generator. Instead of serving dynamic content from a database, it pre-builds HTML files at build time. Fast, secure, simple.

How Eleventy Works

  1. You write content in Markdown
  2. Eleventy reads Markdown files
  3. Eleventy applies templates (Nunjucks) to structure the layout
  4. Eleventy generates HTML files in _site/
  5. GitHub Pages serves those static HTML files

File Organization

Folder/File Purpose
cardiovascular/ Medical specialty folder (creates /cardiovascular/ URL)
heart-failure/index.md Page content (creates /cardiovascular/heart-failure/ URL)
_layouts/ Template files (HTML structure)
_site/ Build output (generated HTML, don't edit)
.eleventy.js Eleventy configuration

Front Matter: Metadata

Every page starts with YAML front matter (between --- lines):

---
title: Atrial Fibrillation
layout: topic.njk
specialty: cardiovascular
description: Diagnosis and management of AF in primary care
---

## Red Flags
Page content goes here...

Eleventy uses this data to:

  • Set the page title
  • Choose which template to use
  • Generate URLs
  • Create navigation and breadcrumbs

Templates (Nunjucks)

Templates are HTML files that wrap your content. A template might look like:

<!-- _layouts/topic.njk -->
<header>
    <h1>{{ title }}</h1>
    <p>{{ description }}</p>
</header>

<main>
    {{ content }}
</main>

<footer>
    By {{ author }}
</footer>

{{ variable }} is a placeholder. Eleventy fills it in with real data from front matter.

Building the Site

Run this in your project folder:

npm run build

This:

  1. Reads all Markdown files
  2. Applies templates
  3. Generates HTML in _site/
  4. Builds the search index
  5. Generates PDFs (optional)

Previewing Locally

npm start

This starts a local server at http://localhost:8080. Pages auto-reload as you edit.

Key insight: You never edit HTML by hand. You edit Markdown, write front matter, and Eleventy generates the HTML. This keeps content separate from design.

Common Issues & Fixes

Build fails with "template not found"

Cause: Typo in layout name (e.g., layout: topci.njk).

Fix: Check spelling in front matter. Available layouts are in _layouts/.

Page doesn't appear in navigation

Cause: Wrong specialty folder name or missing front matter.

Fix: Verify folder name matches front matter specialty field.

Local server doesn't start

Cause: Port 8080 in use, or Node.js not installed.

Fix: Try a different port: eleventy --serve --port 8081.

7. Debugging Workflow

Something will break. When it does, here's how to find and fix it.

The Debugging Mindset

  1. Identify the symptom ("the page doesn't display").
  2. Narrow the cause ("is it HTML, CSS, or the build?").
  3. Fix at the source (don't layer patches).
  4. Verify locally first (don't push broken builds).

Browser Inspector: Your Best Friend

Right-click on any page → Inspect (or press F12).

What to look for:
  • Elements tab: See the HTML structure. Hover over elements to highlight them on the page.
  • Styles panel: See what CSS is applied to the selected element. Green = applied, strikethrough = overridden.
  • Console tab: See JavaScript errors (red text).
  • Network tab: See if images/files fail to load (red status codes like 404).

Common Problems & Solutions

Build fails with "undefined variable"

Error: Template render error
  undefined variable "title" in topic.njk:5

Cause: Front matter is missing a field (e.g., no title).

Fix: Check the file's front matter. Add missing fields.

Page looks wrong in browser

Steps:

  1. Open inspector (F12)
  2. Click the element picker (top-left icon)
  3. Click the broken element on the page
  4. Read the HTML and CSS in inspector
  5. Check the template file for typos

Image doesn't display

In inspector, check:

  • Is the <img> tag there? (HTML problem)
  • Is the src attribute correct? (File path problem)
  • Open the Network tab and look for red 404 errors (file not found)

Styling doesn't apply

In inspector, check:

  • Is the element the right class? (e.g., class="btn-primary")
  • Does the CSS file reference that class? (grep for `.btn-primary`)
  • Is the CSS rule strikethrough? (Another rule overrides it)

Git Debugging

"I broke something, how do I undo?"

# See what changed
git diff

# Undo changes to one file
git checkout filename

# Undo everything
git reset --hard HEAD

"Which commit broke this?"

# See the history
git log --oneline

# See what changed in a specific commit
git show a1b2c3d

Local vs. Live Debugging

Issue Debug Locally If Still Broken
Build fails Read error message, fix file, rebuild Check git history, revert commit
Page looks wrong Inspect browser, check CSS, fix template Rebuild, check _site/ HTML output
Link is broken Check folder structure, verify URL Use inspector Network tab to see 404 errors
Golden rule: Debug locally with the build tool and browser inspector. Don't push to GitHub until you've verified it works locally.

8. Quick Reference

Copy-paste commands and checklists for common tasks.

Setup (First Time)

# Install Node.js dependencies
npm install

# Start local server
npm start

# In browser: http://localhost:8080

Edit a Page

  1. Open the .md file in VS Code
  2. Edit the content or front matter
  3. Save (Ctrl+S)
  4. Browser auto-reloads

Publish Changes

# 1. Build locally
npm run build

# 2. Preview at http://localhost:8080

# 3. Stage changes
git add .

# 4. Commit
git commit -m "your message here"

# 5. Push to GitHub
git push origin main

Create a New Page

# 1. Create folder
mkdir "specialty/new-topic"

# 2. Create file
New-Item "specialty/new-topic/index.md"

# 3. Edit index.md with front matter and content

# 4. Build and test
npm run build
npm start

# 5. Commit and push
git add .
git commit -m "add new-topic page"
git push origin main

Debugging Checklist

  1. ☐ Run npm run build locally. Does it succeed?
  2. ☐ Run npm start and check page in browser.
  3. ☐ Use browser inspector (F12) to check HTML and CSS.
  4. ☐ Check git history: git log --oneline
  5. ☐ If broken, revert last commit: git reset --hard HEAD~1
  6. ☐ Never push if build fails locally.
