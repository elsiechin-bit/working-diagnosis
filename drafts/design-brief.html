Working Diagnosis - Design Brief

Synthesised from design notes  ·  May 2026  ·  Visual impact & clinician scannability

01 - Core principle

The vibe you're building toward

Every decision in these notes orbits one idea: design for panic-search, not leisurely browsing. Doctors are tired, on phones, reheating coffee. Information must be instantly stratifiable.

Doctors panic-search

They don't browse elegantly. They type "leg swelling elderly" or "statin cramps." Search must be prominent, forgiving, and fast - not buried in a nav bar.

GP cognition deserves respect

Use tiered information density. Let a busy GP skim the top layer in 90 seconds, then drill when time allows. Never make them wade through evidence to find the red flag.

"Quietly competent registrar"

Not AI-generated looking. No excessive gradients, giant rounded cards, floating blobs, or startup illustrations. The aesthetic signals depth - the reader trusts it immediately.

04 - Content taxonomy

Content type tag system - implement this first

This is the highest-ROI addition from the notes. Six semantic tag types, each with a distinct colour, let a GP scan a page and immediately know the nature of the information. Pure CSS - no architecture change needed.

Red flags

Symptoms or signs requiring urgent action. Highest visual priority - stands out at a glance. Use sparingly.

Guidelines

NZ MoH, BPAC, HealthPathways, or international guideline-sourced content. Authoritative, citable.

Practical hacks

Clinical shortcuts, mnemonics, "what I actually do." Olive = earthy, practical, battle-tested.

Controversy

Areas of genuine clinical debate, conflicting guidelines, or evolving evidence. Amber = caution, uncertainty.

Medico-legal

Documentation traps, consent issues, ACC considerations. Charcoal = serious, formal weight.

Patient comms

Scripts, framing, explanation strategies. Sage = calm, relational, patient-facing.

CSS implementation Add six utility classes to style.css (.tag-redflags, .tag-guidelines, etc.) using the tokens above. In Markdown content, wrap tags in spans with those classes. For Nunjucks templates, pass a tags array in front matter and render inline. Zero JavaScript needed.

05 - Information architecture

Tiered content - the structural backbone

Three layers of density, always in the same order. A GP on-call can stop at Level 1. A registrar studying for the RNZCGP fellowship reads all three. The structure respects both.

Level 1 - Skim layer The 90-second version ~2 min

At the very top. No paragraphs - only signal. What the busy GP needs before they walk back into the consultation room.

  • Red flags
  • Key threshold changes
  • Diagnostic traps
  • Treatment updates
  • What changed since last year
Level 2 - Practical layer The consultation-room toolkit ~8 min

Flowcharts, dosing tables, specialist referral thresholds, patient communication scripts, clinical pitfalls. What specialists actually care about.

  • Dosing + thresholds
  • Decision flowchart
  • Specialist handover triggers
  • Patient scripts
  • Common pitfalls
Level 3 - Deep dive Evidence + guidelines + controversy deep dive

Journal links, full guideline citations, the "why behind the what," ongoing controversies. For the clinician with time and curiosity.

  • Evidence summary
  • Cochrane / NEJM / BMJ links
  • NZ MoH / BPAC guidelines
  • Controversies + open questions

06 - Search

Persistent search - the most important UX feature

The notes are emphatic: search is everything. It should be prominent, forgiving ("messy retrieval"), and always accessible. Doctors don't search with clinical precision - they search the way they think mid-consultation.

⌕ leg swelling elderly, statin cramps, weird rash child… ⌘K
symptom search medication name acronym (HTN, T2DM) ICD code drug class condition + context
Implementation: Pagefind For an Eleventy static site, Pagefind is the right tool. It generates a static search index at build time, works fully client-side, and has zero server costs. Add the Pagefind build step to your GitHub Actions workflow; add the search UI component to the masthead. The keyboard shortcut (⌘K / Ctrl+K) is the right pattern for clinicians who already live in keyboard-first tools.

07 - Delta learning

"What Changed?" - GPs love delta-learning

One of the most clinically valuable features. GPs don't need to re-read what they already know - they need to know what's different from last time. A structured "What Changed?" format at the top of any updated page delivers this instantly.

Guidelines Hypertension - NZ Management Update Updated Feb 2026
  • Lower treatment threshold: now 130/80 mmHg (down from 140/90) for patients under 65 with CVD risk >10%
  • Ambulatory monitoring preferred over clinic readings for diagnosis in most cases
  • ACEI/ARB: increased caution in frailty + eGFR <30 (previously <45)
Red flags Chest Pain Assessment - BPAC 2025 revision Updated Nov 2025
  • HEART score now recommended over TIMI for primary care risk stratification
  • Point-of-care troponin: acceptable to use if result available within 1 hour of symptom onset
Implementation approach Add a changes array to front matter. The topic layout renders a "What Changed" block at the top of Level 1 when this array is present. No new template needed - a conditional block in topic.njk is enough.

08 - Layout

Recommended: left nav + centre content + right quick panel

The notes recommend a three-column layout. Your current site is already partway there (masthead + main + right sidebar). The main gap is adding a persistent left navigation column for topic-within-pillar wayfinding.

09 - Micro-UX features

Reading time labels, sticky header

Small additions with outsized scannability value for busy clinicians.

Reading time labels

Display estimated reading time for each content tier at the top of the page. Calculated from word count via an Eleventy filter. Options: "2 min skim / 8 min update / deep dive." Sets expectations and respects the reader's time.

2 min skim 8 min update deep dive

Sticky header

The masthead should scroll with the page (position: sticky; top: 0) so navigation and search remain accessible at all times. Currently your masthead is not sticky - a 2-line CSS change. The search bar should live here for instant access.

10 - Implementation status

What's already there vs. what to build

The current site does more of this than you might think. Prioritised below by impact-to-effort ratio.

Feature Effort Notes
Content type tags now CSS + front matter convention Add 6 CSS classes to style.css. Pass tag in front matter; render inline in topic.njk. Highest ROI item - pure CSS, no architecture change.
"What Changed?" block now Front matter + template block Add a changes array to front matter; add a conditional block to topic.njk. Renders at top of page when present. Zero new files.
Sticky header now 2 lines of CSS Add position: sticky; top: 0; z-index: 100; to .masthead. Add a background so it doesn't go transparent on scroll.
Reading time labels now Eleventy filter + front matter Either manually set readtime in front matter, or add a word-count filter in .eleventy.js. Render near the kicker in topic.njk.
Tiered content structure now Content convention + CSS Adopt Level 1/2/3 as a writing convention. Add .tier-1, .tier-2, .tier-3 CSS classes. No new templates needed for basic implementation.
Left in-page nav medium CSS layout change on topic pages Add a third column on wide screens. Generate TOC from headings via JS or Eleventy plugin. Adds significant scannability on long pages.
Search (Pagefind) medium Build step + UI component Add Pagefind to GitHub Actions workflow; add search component to masthead. Fuzzy, client-side, static. The highest-value UX upgrade on the list once basic items are done.
Full palette swap later Global CSS token change Technically a few hours, but disrupts the deployed aesthetic. Not worth doing unless you decide to move away from the current warm editorial direction. Decision needed first.
Font change later Google Fonts swap + CSS tokens Current stack is already distinctive and deployed. Only worth revisiting on a major redesign. Atkinson Hyperlegible is the pick if you ever do this.
Save-to-Toolkit / favourites later localStorage or backend Needs JavaScript. Client-side localStorage works for personal use; requires a backend for cross-device persistence. Tertiary priority - build the content depth first.
Suggested order 1. Content type tags (CSS only, immediate visual impact). 2. Sticky header. 3. "What Changed?" blocks on a few high-traffic pages. 4. Reading time labels. 5. Search (Pagefind) - this is the biggest lift but also the biggest payoff for the clinical audience.
